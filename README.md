
**Project Sentinel: Rebuilding AML STR Submission as an AI-Native System**

Project Sentinel is not a workflow enhancement tool, it is a re-architecture of how Suspicious Transaction Reports (STRs) are created, validated, and submitted. Traditional STR processes are human-linear: data is collected, reviewed manually, documented in static templates, and finally entered into regulator portals. AI, when added to this system, often becomes a summarization layer bolted onto legacy steps. Sentinel takes a different approach: it treats the STR itself as a machine-generated artifact born from structured data, adversarial reasoning, and schema-level automation.

At its core, Sentinel ingests raw case inputs - KYC records, alert data, and transaction histories — as structured datasets. Instead of asking an analyst to manually interpret and narrate risk, the system deploys an adversarial multi-agent architecture. A “Prosecutor” agent aggressively identifies red flags such as structuring, behavioral inconsistencies, and economic mismatches. A “Red Team” agent challenges those assumptions, surfacing legitimate explanations and testing bias. A “Narrator” agent then synthesizes both perspectives into an objective, regulator-ready risk brief. This transforms analysis from a single-threaded human judgment into a structured, debate-driven reasoning process.

Crucially, the output of this reasoning engine is not a Word document or PDF. It is structured JSON aligned directly to regulatory schema requirements. The system maps investigative findings into a FINTRAC-compliant schema, enforcing structural integrity at generation time. Validation and formatting are not afterthoughts; they are embedded in the generation layer. By constraining AI outputs to strict JSON schemas, Sentinel eliminates manual transcription errors and reduces compliance risk.

The human role is also redesigned. Instead of writing narratives from scratch, analysts operate as adjudicators and editors. They review AI reasoning across structured tabs, inspect the debate, and refine the final JSON draft through an embedded editor with real-time validation. The workflow becomes supervisory rather than clerical. This shifts analyst time from document preparation to critical judgment and quality control. Sentinel also treats state as dynamic. The application tracks case progression — ingestion, adversarial analysis, adjudication, and submission, within a unified interface. Submission readiness is determined by structural validation rather than manual checklist confirmation. This makes compliance measurable and programmable.

By rebuilding STR filing as an AI-native pipeline, Sentinel removes redundant manual steps, reduces cognitive bias through adversarial reasoning, and embeds regulatory compliance directly into system logic. It is not “AI-assisted reporting.” It is reporting generated, structured, and validated by AI from inception with humans acting as strategic overseers. In doing so, Project Sentinel demonstrates what compliance looks like when designed for AI from the ground up: structured inputs, machine-mediated reasoning, schema-bound outputs, and human-in-the-loop governance.
